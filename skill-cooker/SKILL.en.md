# skill-cooker

A fully automated pipeline that turns natural-language requirements into standardized skills.
The name comes from how it works: you supply the ingredients (a requirement description),
and it follows the standard recipe (skill-creator spec) through two-draft generation,
blind comparison, and iterative testing — until a thoroughly validated skill is ready to serve.

**Purpose**: A supervised acceleration tool, not a black box that replaces human judgment.
Two human confirmation checkpoints are built into the flow; you hold final decision authority.

---

## Step 0: Environment Detection and Mode Declaration (runs automatically)

skill-cooker detects the runtime environment and announces its operating mode before
the workflow begins.

### Mode selection logic

```
if current environment is Claude Code CLI:
    check whether skill-creator path is accessible:
    → accessible:     Full Mode
    → not accessible: report missing path and abort
elif current environment is Claude.ai web:
    → Degraded Mode — explain limitations to user, then continue
else:
    → no filesystem available; abort immediately and explain why
```

### Full Mode

Subagents provide genuinely blank context windows, so cognitive isolation between roles
is complete and effective. All 9 design constraints are fully enforced, and the comparison
between tmp_claude and tmp_standard carries its full intended value.

### Degraded Mode ⚠️

**When running on Claude.ai, output the following declaration before continuing:**

```
⚠️ Degraded Mode Notice

The current environment (Claude.ai) cannot create independent blank context windows
(subagents). The two-draft comparison at the heart of skill-cooker has a structural
limitation in this mode:

- tmp_claude and tmp_standard are both generated by the same Claude instance
- When generating tmp_standard, Claude already knows the content of tmp_claude
- Differences between the two versions reflect format transformation, not independent understanding
- The comparative analysis has limited reference value

You can continue, but output quality will be lower than Full Mode.
Running skill-cooker inside Claude Code is recommended for full effectiveness.

Continue with the above limitations in mind? [Yes / No]
```

If the user chooses No, abort. If Yes, continue — all subsequent steps run in
Degraded Mode (Roles I / II / III execute inline serially with no real context isolation).

## Overall Flow

```
                          Full Mode               Degraded Mode (Claude.ai)
[Step 1] Understand req   subagent Role I          inline, stored in conversation context
              ↓ pre-check passes
[Step 2] Generate std     subagent Role II         inline + forced-forgetting declaration
              ↓
[Step 3] Compare+test     roles parallel + grader  inline serial, no real isolation
              ↓ ← human confirmation checkpoint (same in both modes)
         convergence check → final human confirmation
              ↓
[Step 4] Deliver          bash file ops            present_files download
```

## Step 1: Understand Requirements and Generate tmp_claude

**Executed by Role I (Pure-Requirement Generator).** Role I's context must not contain
any skill-creator content — the value of tmp_claude lies precisely in its intuitive,
uncontaminated understanding of the requirement.

Role I reads the user's requirement and checks whether the following four dimensions
are sufficiently specified:

- **Trigger scenario** — In what situation would a user invoke this skill, and with what phrasing?
- **Core behavior** — What does the skill do, and in how many steps?
- **Output format** — Does it produce a file, conversational text, or an action?
- **Target user** — Who uses this skill, and what is their technical background?

**Dimension sufficiency rules:**

- Dimension **explicitly provided**: use it directly, no need to ask
- Dimension **reliably inferable from context**: skip asking, but state the inference to the user
  before generating tmp_claude (e.g., "I'll treat the target user as a developer with Python
  experience — let me know if that's wrong")
- Dimension **entirely missing or too vague to infer**: ask the user and wait for a response

The inference statement must be output before any skill content is generated,
giving the user a chance to correct it.

Once information is sufficient (or inferences are confirmed), generate
`skill-cooker-workspace/tmp_claude/SKILL.md`.

**tmp_claude minimum viability pre-check** (run by the main flow, not Role I):

```
✓ frontmatter contains a non-empty name field
✓ frontmatter contains a non-empty description field
✓ body contains at least one identifiable behavioral step
```

If pre-check fails, return to Step 1 for Role I to fix — do not proceed to Step 2.

🔸 **Degraded Mode**: Role I executes inline (same Claude instance).
tmp_claude content stays in the conversation context; no filesystem write needed.
Pre-check runs inline by the main flow using the same criteria.

## Step 2: Generate tmp_standard

**Executed by Role II (Standard Builder), first invocation.**

Role II follows the skill-creator **"creating from scratch"** branch. Its input is the
user's original requirement text. **tmp_claude's full content is not passed in** —
this prevents anchoring effects from undermining the comparison value of the two drafts.
Role II must read skill-creator/SKILL.md and follow its spec strictly.

Output: `skill-cooker-workspace/tmp_standard/SKILL.md`

Full context boundary rules for all roles: see `references/roles.md`.

🔸 **Degraded Mode**: Role II executes inline. Before generating, it must output the
following forced-forgetting declaration and then follow it strictly — no reviewing or
referencing any tmp_claude content:

```
[Forced-Forgetting Declaration]
I am now fully ignoring all tmp_claude content in my context.
The following tmp_standard is generated solely from the user's original requirement
and the skill-creator spec, without referencing any prior draft.
User requirement: [original requirement text]
```

After outputting the declaration, immediately generate following skill-creator's
"creating from scratch" branch. tmp_standard content stays in the conversation context.

## Step 3: Compare, Test, Iterate

The main flow coordinates three roles through three stages:

### Sequencing of Stages 3-A and 3-B

**Claude Code (with subagents)**: 3-A and 3-B can be launched in parallel — blind
comparison and eval design are independent of each other. Launching both simultaneously
saves time. The main flow waits for both to complete before entering the 3-C iteration loop.

**Claude.ai (no subagents)**: execute serially — complete 3-A first, then 3-B.

### Stage 3-A: Blind Comparison (executed by Role III)

The main flow randomly labels tmp_claude and tmp_standard as "Version X" and "Version Y"
and passes them to Role III for blind evaluation. Role III does not know which version
was produced by whom.

Evaluation dimensions (passed in by the main flow):
- Does the trigger description cover the core use cases?
- Is the step logic complete and in a sensible order?
- Are context boundaries clear (for skills that use subagents)?
- Is the output format unambiguous?
- Are there unhandled edge cases?

Role III outputs a structured diff summary. After unblinding, the main flow **lists
any content that was removed or weakened** separately, and asks the user to confirm
whether those removals are appropriate before proceeding.

🔸 **Degraded Mode**: genuine blind evaluation is impossible (the same Claude knows
the identity of both versions). Instead, output a declaration and then analyze inline:

```
[Blind Evaluation Declaration]
I will analyze the following two versions as objectively as possible,
without favoring the one I generated.
```

Output the structured diff summary using the evaluation dimensions above.
Removed or weakened content must still be listed separately for user confirmation.
The reference value of this stage is limited — apply your own judgment when confirming.

### Stage 3-B: Dynamic Eval Design (Role I, second invocation — fresh instance)

At this point Role I is still in a pure "knows only the user requirement" state,
with no exposure to any skill implementation. Assertions designed this way are based on
the **requirement specification**, not the implementation, which produces more uniform coverage.

Requirements:
- At least 5 assertions (force supplementation if fewer)
- At least 2 assertions specifically targeting edge cases and failure modes
- Each assertion description must state which known failure mode it targets

Output: `skill-cooker-workspace/evals/evals.json` (format: see `references/eval-schema.md`)

🔸 **Degraded Mode**: Role I executes inline. Before designing evals, output:

```
[Eval Design Declaration]
The following assertions are designed based solely on the user's original requirement.
I am ignoring the specific implementations in tmp_claude and tmp_standard.
```

evals.json content stays in the conversation context; no filesystem write needed.

### Stage 3-C: Test Iteration Loop

Each iteration:

1. **Run tests**: Score each assertion against tmp_standard using a separate Role III
   instance per assertion (fully isolated across assertions and test cases
   to prevent baseline drift).

   🔸 **Degraded Mode**: score inline one assertion at a time. Before each assertion, output:
   ```
   [Scoring Isolation Declaration — Assertion N of Total]
   Judging only whether tmp_standard satisfies this assertion,
   independent of all other assertions.
   ```

2. **Aggregate results**: the main flow computes the pass rate and records this round's data.

3. **Human confirmation checkpoint**: the main flow presents:
   ```
   ─────────────────────────────────
   Round N Test Results
   Pass rate: X%  (previous: Y%,  improvement: +Z%)
   3 weakest assertions: [list]
   Structured improvement summary: [list]
   ─────────────────────────────────
   Choose: [Continue iterating]  [Accept current version]  [View detailed report]
   ```

4. **User chooses to continue**: the main flow distills "failed assertions + comparison
   summary + user feedback" into improvement instructions and passes them to a fresh
   Role II instance, which produces an improved tmp_standard.

**Convergence condition** (both conditions required — AND, not OR):
- Pass rate ≥ 85%
- Improvement in each of the last two consecutive rounds < 5%

⚠️ **Rounds 1 and 2 use a different convergence check than Round 3 onward.**
The first two rounds lack enough data to evaluate "two consecutive rounds of improvement",
so only the pass rate threshold is checked. Both conditions are checked from Round 3 onward.
Read `references/convergence.md` when aggregating each round's results — required every round.

**Local optimum handling**: if improvement stalls (both recent rounds < 5%) but pass rate
is still below 85%, stop iterating and report: "Local optimum reached (current pass rate X%).
Manual intervention or accepting the current version is recommended."

**Hard cap**: maximum 5 iterations. After Round 5, regardless of pass rate, report the
current state and ask the user to decide.

**Improvement priority**: user feedback > automatically inferred improvement direction.

---

## Step 4: Clean Up and Deliver

**Final human confirmation checkpoint after convergence.** Present:
- Full text of the final tmp_standard
- Complete test report (pass-rate curve across all rounds + final assertion states)

After the user confirms, ask for the target skill name and execute the following
(atomicity guaranteed):

```bash
# 1. Check whether the target name already exists (conflict → report error, wait for new name)
# 2. Copy tmp_standard to the target path
# 3. Verify backup file integrity
# 4. On successful verification, delete the tmp_claude directory
# 5. Delete the tmp_standard directory
# 6. Report successful delivery
```

If any step fails, retain all temporary files, report where the failure occurred,
and do not execute subsequent steps.

🔸 **Degraded Mode**: no bash execution available. Use the following delivery flow instead:

1. Ask for the final skill name
2. Write tmp_standard content to `/mnt/user-data/outputs/<target-name>/SKILL.md`
3. Call `present_files` to surface the file for the user to download
4. Tell the user to place the downloaded file in one of the following paths to install:
   - `~/.claude/skills/<target-name>/SKILL.md` (global install)
   - `.claude/skills/<target-name>/SKILL.md` (project-level install)
5. Temporary content (tmp_claude, tmp_standard, evals) in the conversation context
   ends naturally with the conversation — no explicit cleanup needed
