# skill-cooker

把自然语言需求转化为标准化 skill 的全自动流水线。名字来自于它的工作方式：
你提供食材（需求描述），它按照标准食谱（skill-creator 规范）经过两阶段草稿、
盲评对比、迭代测试，最终端出一个经过充分检验的成品 skill。

**定位**：有监督的加速工具，不是替代人类判断的黑箱。流程中有两个人工确认节点，
你始终掌握最终决策权。

---

## 第零步：环境检测与模式声明（自动执行）

skill-cooker 根据运行环境自动判断工作模式，并在流程开始前向用户明确声明。

### 模式判断逻辑

```
if 当前环境为 Claude Code CLI:
    检查 skill-creator 路径是否可访问：
    → 可访问：完整模式（Full Mode）
    → 不可访问：报告缺失路径，中止
elif 当前环境为 Claude.ai 网页版:
    → 降级模式（Degraded Mode），向用户说明局限后继续
else:
    → 无文件系统，直接中止并说明原因
```

### 完整模式（Full Mode）

subagent 提供真正的空白上下文窗口，角色间认知隔离完整有效。
所有 9 条设计约束完整生效，tmp_claude 与 tmp_standard 的对比具有完整价值。

### 降级模式（Degraded Mode）⚠️

**在 Claude.ai 中，向用户输出以下声明后才能继续：**

```
⚠️ 降级模式说明

当前环境（Claude.ai）无法创建独立的空白上下文窗口（subagent）。
skill-cooker 的两阶段草稿对比在此模式下存在结构性局限：

- tmp_claude 和 tmp_standard 由同一个 Claude 实例生成
- 生成 tmp_standard 时，Claude 已知晓 tmp_claude 的内容
- 两个版本的差异反映「格式变换」而非「独立理解」
- 对比分析的参考价值有限

可以继续运行，但产出质量低于完整模式。
建议在 Claude Code 中使用 skill-cooker 以获得完整效果。

是否在了解上述局限的情况下继续？[是 / 否]
```

用户选择「否」则中止；选择「是」则继续，后续步骤按降级模式执行
（角色 I / II / III 改为串行 inline 执行，无真实上下文隔离）。

## 整体流程

```
                        Full Mode          Lite Mode（Claude.ai）
[第一步] 需求理解 + 交互  subagent 角色 I    inline，对话上下文存储
              ↓ 预检通过
[第二步] 生成 standard   subagent 角色 II   inline + 强制遗忘指令
              ↓
[第三步] 对比+测试+迭代   角色并行+grader    inline 串行，无真实隔离
              ↓ ← 人工确认节点（两种模式相同）
         收敛判断 → 人工最终确认
              ↓
[第四步] 交付             bash 文件操作      present_files 下载
```

---

## 第一步：需求理解与 tmp_claude 生成

**由角色 I（需求纯净生成者）执行。** 角色 I 的上下文中禁止出现 skill-creator 的任何内容——
tmp_claude 的价值正在于它对需求的未被规范污染的直觉理解。

角色 I 读取用户需求后，判断以下维度是否充分：

- 触发场景（用户会在什么情况下说什么话来启动这个 skill？）
- 核心行为（skill 做什么？分几步？）
- 输出格式（产出物是文件、对话内容、还是操作？）
- 目标用户（谁用这个 skill？技术背景如何？）

**维度充分性判断规则**：

- 维度**明确提供**：直接使用，无需追问
- 维度**可从上下文可靠推断**：跳过追问，但在生成 tmp_claude 前向用户说明推断内容
  （例：「我将目标用户理解为有 Python 基础的开发者，如有不同请告知」）
- 维度**完全缺失或过于模糊无法推断**：主动追问，等待补充后继续

推断说明必须在生成任何 skill 内容之前输出，让用户有机会纠正，再继续。

信息充分（或推断已确认）后，生成 `skill-cooker-workspace/tmp_claude/SKILL.md`。

**tmp_claude 最低可执行性预检**（由主流程执行，角色 I 不参与）：

```
✓ frontmatter 中有 name 字段
✓ frontmatter 中有 description 字段（非空）
✓ 正文中有至少一个可识别的行为描述步骤
```

预检不通过，返回第一步让角色 I 修复，不进入第二步。

🔸 **降级模式**：角色 I 改为 inline 执行（同一 Claude 实例）。
tmp_claude 内容保留在对话上下文中，无需写入文件系统。
预检由主流程 inline 完成，标准不变。

---

## 第二步：生成 tmp_standard

**由角色 II（标准构建者）首次调用执行。**

角色 II 走 skill-creator 的 **"creating from scratch"** 分支，输入是用户需求原文，
**不传入 tmp_claude 全文**（防止锚定效应损害两个版本的对比价值）。
角色 II 需读取 skill-creator/SKILL.md 并严格遵循其规范。

产出：`skill-cooker-workspace/tmp_standard/SKILL.md`

各角色的详细上下文边界见 `references/roles.md`。

🔸 **降级模式**：角色 II 改为 inline 执行，在生成前必须输出以下强制遗忘指令，
然后严格按指令执行，不得回顾或引用 tmp_claude 的任何内容：

```
【强制遗忘声明】
我现在将完全忽略上下文中 tmp_claude 的所有内容。
以下 tmp_standard 仅基于用户原始需求和 skill-creator 规范生成，
不参考任何已有草稿。
用户需求：[需求原文]
```

输出强制遗忘声明后，立即按 skill-creator "creating from scratch" 分支生成。
tmp_standard 内容保留在对话上下文中，无需写入文件系统。

---

## 第三步：对比、测试、迭代

这一步由主流程协调三个角色协同完成，分为三个阶段：

### 阶段 3-A 与 3-B 的时序

**Claude Code（有 subagent）**：3-A 和 3-B 可以并行启动——盲评对比和 eval 设计
互不依赖，同时发起可节省时间。主流程等两者都完成后，再进入 3-C 迭代循环。

**Claude.ai（无 subagent）**：串行执行，先完成 3-A，再执行 3-B。

### 阶段 3-A：盲评对比（角色 III 执行）

主流程将 tmp_claude 和 tmp_standard 随机标记为"版本 X"和"版本 Y"，
传给角色 III 进行盲评。角色 III 不知道哪个版本是谁生成的。

评估维度清单（主流程传入）：
- 触发描述是否覆盖核心使用场景
- 步骤逻辑是否完整、顺序是否合理
- 上下文边界是否清晰（对于含 subagent 的 skill）
- 输出格式是否明确
- 是否有未处理的边缘情况

角色 III 输出结构化差异摘要，主流程去盲后，将摘要中**"被删除或弱化的内容"单独列出**，
向用户确认这些删除是否合理，再进入下一阶段。

🔸 **降级模式**：无法执行真正的盲评（同一 Claude 知道两个版本的身份）。
改为输出强制遗忘声明后 inline 分析：

```
【盲评声明】
我将尽量客观地分析以下两个版本，不因自己生成了其中一个而偏袒它。
```

然后按评估维度清单输出结构化差异摘要，"被删除或弱化的内容"仍需单独列出请用户确认。
此阶段的参考价值有限，用户确认时请结合自身判断。

### 阶段 3-B：动态 eval 设计（角色 I 第二次调用，全新实例）

此时角色 I 仍处于"只知用户需求"的纯净状态，不接触任何 skill 实现。
这样设计出来的 assertions 基于**需求规格**而不是**实现方式**，覆盖更均匀。

要求：
- 至少 5 条 assertions（少于 5 条则强制补充）
- 至少 2 条专门针对边缘情况和失败模式的 assertions
- 每条 assertion 描述中说明"这个检查项针对哪种已知失败模式"

产出：`skill-cooker-workspace/evals/evals.json`（格式见 `references/eval-schema.md`）

🔸 **降级模式**：角色 I 改为 inline 执行，在设计 eval 前输出强制遗忘声明：

```
【Eval 设计声明】
以下 assertions 仅基于用户原始需求设计，
我将忽略 tmp_claude 和 tmp_standard 的具体实现内容。
```

evals.json 内容保留在对话上下文中，无需写入文件系统。

### 阶段 3-C：测试迭代循环

每轮迭代流程：

1. **运行测试**：对 tmp_standard 的每条 assertion，角色 III 独立实例打分
   （不同 assertion、不同测试用例之间完全隔离，防止基准漂移）

   🔸 **降级模式**：改为 inline 逐条打分，每条 assertion 前输出：
   ```
   【评分隔离声明 - Assertion N/总数】
   仅根据 tmp_standard 对本条 assertion 的满足情况判断，不受其他条目影响。
   ```

2. **聚合结果**：主流程计算通过率，记录本轮数据

3. **人工确认节点**：主流程向用户展示：
   ```
   ─────────────────────────────────
   第 N 轮测试结果
   通过率：X%（上轮：Y%，改善：+Z%）
   最薄弱的 3 个 assertions：[列出]
   结构化改进摘要：[列出]
   ─────────────────────────────────
   选择：[继续迭代] [接受当前版本] [查看详细报告]
   ```

4. **用户选择继续**：主流程将"失败 assertions + 对比摘要 + 用户反馈"
   提炼为改进指令，传给角色 II（全新实例），产出改进版 tmp_standard

**收敛条件**（双条件 AND，缺一不可）：
- 通过率 ≥ 85%
- 连续两轮改善幅度均 < 5%

⚠️ **第 1、2 轮的收敛判断与第 3 轮起不同**：前两轮数据不足以判断"连续两轮改善"，
只检查通过率是否达标。第 3 轮起才同时检查双条件。
完整计算逻辑见 `references/convergence.md`，每轮聚合结果时必须读取。

**局部最优处理**：改善停滞（连续两轮 < 5%）但通过率未达 85% 时，
不继续迭代，向用户报告："已达局部最优（当前通过率 X%），建议人工介入或接受当前结果。"

**硬上限**：最多 5 轮迭代，超出后无论通过率如何，报告当前状态，请用户决策。

**改进优先级**：用户反馈 > 自动推导的改进方向。

---

## 第四步：清理与交付

**收敛后最终人工确认节点**，展示：
- tmp_standard 最终版本全文
- 完整测试报告（所有轮次通过率曲线 + 最终 assertions 状态）

用户确认后，询问最终 skill 的目标名称，执行以下操作（原子性保证）：

```bash
# 1. 检查目标名称是否已存在（冲突则报错，等待用户重新指定）
# 2. 备份：将 tmp_standard 复制到目标路径
# 3. 验证备份文件完整性
# 4. 验证成功后，删除 tmp_claude 目录
# 5. 删除 tmp_standard 目录
# 6. 报告交付成功
```

任何步骤失败，保留所有临时文件，报告失败位置，不继续执行后续步骤。

🔸 **降级模式**：无 bash 执行能力，改为以下交付流程：

1. 询问最终 skill 名称
2. 将 tmp_standard 内容写入 `/mnt/user-data/outputs/<目标名称>/SKILL.md`
3. 调用 `present_files` 工具，将文件呈现给用户下载
4. 告知用户将下载的文件放入以下路径之一完成安装：
   - `~/.claude/skills/<目标名称>/SKILL.md`（全局安装）
   - `.claude/skills/<目标名称>/SKILL.md`（项目级安装）
5. 对话上下文中的 tmp_claude、tmp_standard、evals 临时内容
   已随对话自然结束，无需额外清理
