# 收敛判断完整逻辑

## Assertions 集合锁定规则

收敛判断的数学前提是：**每轮使用完全相同的 assertions 集合**。
通过率的「改善幅度」只有在分母（assertion 总数）不变时才有意义。

**锁定时机**：阶段 3-B 完成、evals.json 生成后，assertions 集合立即锁定。
此后整个迭代过程中，assertions 集合不允许修改或增删。

**如果用户在迭代中途要求修改 assertions**：
1. 接受修改，更新 evals.json
2. **迭代轮次计数清零**，从第 1 轮重新开始
3. 告知用户：「Assertions 已更新，迭代从第1轮重新计算」

不清零直接继续会导致改善幅度比较失去意义，可能产生错误的收敛判断。

---



```json
{
  "round": 1,
  "pass_count": 3,
  "total_count": 5,
  "pass_rate": 0.60,
  "improvement_over_previous": null
}
```

第一轮的 `improvement_over_previous` 为 null（无上一轮数据）。

---

## 收敛条件（双条件 AND）

```
收敛 = (pass_rate >= 0.85) AND (连续两轮 improvement < 0.05)
```

**两个条件都必须满足，缺一不可。**

下面列出所有组合情况：

| pass_rate | 连续两轮改善 | 判断 | 行动 |
|-----------|------------|------|------|
| ≥ 85% | 均 < 5% | ✅ 收敛 | 进入最终人工确认节点 |
| ≥ 85% | 任一 ≥ 5% | ⏳ 未收敛 | 继续迭代（仍在改善中） |
| < 85% | 均 < 5% | ⚠️ 局部最优 | 报告并请用户决策（见下） |
| < 85% | 任一 ≥ 5% | ⏳ 未收敛 | 继续迭代 |

---

## 局部最优处理

当 pass_rate < 85% 且连续两轮改善均 < 5% 时，向用户展示：

```
⚠️ 已达局部最优

当前通过率：X%（目标：85%）
最近两轮改善：+Y%、+Z%（均低于 5% 阈值）

skill-cooker 认为继续自动迭代改善空间有限。
可能的原因：
- 失败的 assertions 需要对 skill 做结构性修改，超出当前迭代能力
- 部分 assertions 设计过于严格，可能需要调整
- 需要引入用户领域知识才能突破

选择：
[A] 查看所有失败的 assertions，人工指导下一步方向
[B] 放宽收敛目标，接受当前版本（通过率 X%）
[C] 重新设计 eval assertions（可能部分 assertions 不合适）
```

---

## 硬上限保护

最多执行 5 轮迭代。第 5 轮结束后，无论通过率如何，展示：

```
已完成 5 轮迭代（硬上限）

迭代历史：
第1轮：X1%
第2轮：X2%（+Y2%）
第3轮：X3%（+Y3%）
第4轮：X4%（+Y4%）
第5轮：X5%（+Y5%）

当前通过率：X5%

选择：
[A] 接受当前版本（推荐，如通过率 ≥ 70%）
[B] 手动继续迭代（主流程暂停，由用户直接修改 tmp_standard）
[C] 重新开始（保留 evals，重新生成 tmp_standard）
```

---

## 改善幅度计算

```
improvement(N) = pass_rate(N) - pass_rate(N-1)
```

「连续两轮改善均 < 5%」的判断需要至少 3 轮数据（第1轮无改善数据，
第2轮有一轮，第3轮起才能判断"连续两轮"）。

因此：
- 第 1、2 轮：只看 pass_rate 是否达标，不做连续改善判断
- 第 3 轮起：同时判断两个收敛条件

---

## 通过率的统计有效性保护

当 total_count（assertion 总数）< 5 时，85% 的通过率阈值没有统计意义
（5 条中 4 条通过 = 80%，差距只有 1 条）。

此时不进入迭代，返回阶段 3-B，强制要求角色 I 补充 assertions 至 5 条以上。
